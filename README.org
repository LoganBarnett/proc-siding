#+title:     proc-siding
#+language:  en

* Introduction

proc-siding is a general-purpose pressure monitor for safe-idle-worker
governance.  It watches a resource for sustained external pressure and triggers
configurable actions when that pressure crosses a threshold — then reverses
those actions once the pressure clears.

proc-siding is not a detection mechanism — it is a trigger.  Detection is
handled by a user-configured command (the *detector*) that runs at a regular
interval and writes tab-separated pressure readings to stdout.  proc-siding
sums the values, applies hysteresis, and fires actions on state transitions.
The detector is responsible for all resource-specific logic, filtering, and
exclusion.

The motivating use case is GPU arbitration: a garage-queue worker runs large
language model inference on a shared machine.  When a game or other GPU-heavy
process launches, proc-siding pauses the worker so the GPU is fully available,
then resumes it automatically once the external load is gone.  But the core
mechanism is resource-agnostic — any detector that produces a numeric pressure
value can drive it.

Hysteresis prevents flapping: N consecutive above-threshold samples are
required before pausing, and N consecutive below-threshold samples before
resuming.

* Installation

** From source

Requires a Rust toolchain (stable, 1.78+).

#+begin_src sh
git clone <repo-url> proc-siding
cd proc-siding
cargo build --release
# binary at target/release/proc-siding
#+end_src

** Nix

The flake exports ~nixosModules.default~ (systemd service, runs as root) and
~darwinModules.default~ (launchd user agent).  Add the flake as an input and
import the appropriate module, then enable the service and supply settings:

#+begin_src nix
services.proc-siding = {
  enable = true;
  settings = {
    pressure = { threshold = 25; hysteresis = 3; poll_interval_ms = 2000; };
    detector_cmd = "examples/detectors/amd-gpu.sh --exclude-unit ollama.service";
    action = {
      kind = "http_post";
      pressure_url = "http://127.0.0.1:9091/control/pause";
      clear_url    = "http://127.0.0.1:9091/control/resume";
    };
  };
};
#+end_src

* Usage

#+begin_src sh
proc-siding --config /etc/proc-siding/config.toml
#+end_src

Log level and format are controlled via environment variables:

#+begin_src sh
RUST_LOG=debug proc-siding --config config.toml   # verbose per-tick output
RUST_LOG=info  proc-siding --config config.toml   # transitions only (default)
#+end_src

At ~INFO~ level the service is quiet except on state changes:

#+begin_example
INFO proc_siding_lib::monitor: Starting pressure monitor threshold=25.0 hysteresis=3 poll_interval_ms=2000
INFO proc_siding_lib::monitor: Pressure sustained; triggering actions pressure=45.0 contributors="glmark2-es2-wayland"
INFO proc_siding_lib::monitor: Pressure cleared; triggering actions pressure=8.0
#+end_example

At ~DEBUG~ level every poll tick is logged, including the current contributor
list and hysteresis counters.

* Configuration

All settings live in a single TOML file passed via ~--config~.

#+begin_src toml
[pressure]
threshold        = 25     # percent, 0–100; default 25
hysteresis       = 3      # consecutive samples before state change; default 3
poll_interval_ms = 2000   # default 2000

# Command to run as the resource poller.  Must output TSV lines of
# <value><TAB><entity> on stdout.  Exit code 0 means a successful sample.
detector_cmd = "examples/detectors/amd-gpu.sh --exclude-unit ollama.service"

[action]
kind         = "http_post"                          # "http_post" | "http" | "exec"
pressure_url = "http://127.0.0.1:9091/control/pause"
clear_url    = "http://127.0.0.1:9091/control/resume"
# For exec:
# kind         = "exec"
# pressure_cmd = "systemctl stop my-worker"
# clear_cmd    = "systemctl start my-worker"
#+end_src

** Actions

| ~kind~       | Description                                     |
|--------------+-------------------------------------------------|
| ~http_post~  | Sends HTTP POST to ~pressure_url~ / ~clear_url~ |
| ~http~       | Configurable HTTP method per transition          |
| ~exec~       | Runs a shell command per transition              |

The ~http~ action supports configurable methods and Unix socket transport:

#+begin_src toml
[action]
kind            = "http"
pressure_url    = "unix:/run/my-service/control.sock:/mute"
pressure_method = "PUT"
clear_url       = "unix:/run/my-service/control.sock:/unmute"
clear_method    = "DELETE"
#+end_src

Additional actions can run alongside the primary one:

#+begin_src toml
[[extra_actions]]
kind         = "exec"
pressure_cmd = "notify-send 'GPU pressure detected'"
clear_cmd    = "notify-send 'GPU pressure cleared'"
#+end_src

** Detectors

A detector is any executable that writes TSV rows to stdout.  Each row carries
a numeric pressure value and a human-readable entity label:

#+begin_example
42.5	firefox
15.0	blender
#+end_example

proc-siding sums the values to produce an aggregate pressure reading and uses
the entity names for log output only.  Exit code 0 signals a successful sample;
any non-zero exit causes the tick to be skipped.

See ~examples/detectors/~ for ready-to-use GPU detector scripts covering AMD,
NVIDIA, and Metal.

** Metrics

An optional Prometheus metrics endpoint can be enabled:

#+begin_src toml
metrics_listen = "127.0.0.1:9191"
#+end_src

Exposed metrics:

| Metric                                     | Type    | Description                       |
|--------------------------------------------+---------+-----------------------------------|
| ~proc_siding_pressure_transitions_total~   | counter | Times pressure triggered actions  |
| ~proc_siding_clear_transitions_total~      | counter | Times pressure cleared            |
| ~proc_siding_action_errors_total~          | counter | Action failures                   |
| ~proc_siding_pressure_sample~              | gauge   | Most recent aggregate pressure    |

* Gotchas and Tradeoffs

- *Detector privilege depends on the script.*  The AMD and NVIDIA example
  detectors read ~/proc/<pid>/fd~ for processes they do not own, which requires
  ~CAP_SYS_PTRACE~ or running as root.  The Metal detector runs at user level.

- *GPU utilisation is not attributed per-process on Linux.*  The AMD and
  NVIDIA example detectors report the overall GPU busy percentage (from sysfs or
  nvidia-smi) whenever any external process has the device open.  If multiple
  external processes share the GPU, the reported figure is the total — there
  is no per-process split.  The detector scripts divide the total evenly across
  contributors as an approximation.

- *The Metal detector blocks for ~interval_ms~ per poll.*  Each call to
  ~metalps~ takes one sample interval to complete, so effective poll latency
  is ~poll_interval_ms + interval_ms~.

- *Kernel truncates ~/proc/<pid>/comm~ at 15 characters.*  The contributor
  name in log output may be truncated for processes with long names.

- *Hysteresis applies symmetrically.*  The same counter threshold governs both
  the pause and the resume.  There is no way to configure asymmetric
  hysteresis (e.g. fast-pause / slow-resume) without changing the source.

* How to Exercise GPU Pressure

** Linux (AMD / NVIDIA)

~glmark2~ is included in ~environment.systemPackages~ when the service is
enabled, so it is available on any host running proc-siding.

#+begin_src sh
XDG_RUNTIME_DIR=/run/user/$(id -u) WAYLAND_DISPLAY=wayland-0 \
  glmark2-es2-wayland &
journalctl -fu proc-siding
#+end_src

Watch for ~Pressure sustained~ in the log after roughly
~hysteresis × poll_interval_ms~ milliseconds (~6 s~ at defaults), then
~Pressure cleared~ after glmark2 exits.

** macOS (Metal)

~mlx-metal-test~ is included in ~environment.systemPackages~ when the service
is enabled.

#+begin_src sh
mlx-metal-test &
tail -f /tmp/proc-siding.log
#+end_src

Or stream logs via Console.app / ~log stream~.  The log file path is
configured via ~services.proc-siding.logFile~ (default ~/tmp/proc-siding.log~).

** Debug logging

Set ~RUST_LOG=debug~ (in the environment or in the systemd unit's
~Environment=~ / launchd ~EnvironmentVariables~) to see every poll tick with
contributor names and hysteresis counter values.

* Contributing

Run the test suite with ~cargo test --workspace~.  Format with ~cargo fmt~ and
lint with ~cargo clippy~ before submitting changes.

The crate structure keeps all business logic in ~crates/lib~ and limits
~crates/cli~ to wiring.  New action implementations belong in
~crates/lib/src/action.rs~ with a corresponding variant added to
~crates/lib/src/config.rs~ and wired in ~crates/cli/src/main.rs~.

New detectors are standalone scripts in ~examples/detectors/~.  See
~examples/detectors/README.org~ for the output contract.
