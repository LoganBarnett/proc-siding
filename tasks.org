* DONE Track and expose Prometheus metrics for pressure transitions and action failures

Forward-port the Prometheus exporter pattern from ~rust-template~.  The template
uses the ~prometheus~ crate (0.13) with ~Registry~, ~IntCounter~, and
~TextEncoder~.

proc-siding is not a web server, so expose metrics via a minimal HTTP listener
on a configurable address (e.g. ~metrics_listen = "127.0.0.1:9191"~; disabled
when absent).

Metrics to expose:

- ~proc_siding_pressure_transitions_total~ (counter) — times pressure was
  detected and actions fired.
- ~proc_siding_clear_transitions_total~ (counter) — times pressure cleared and
  actions fired.
- ~proc_siding_action_errors_total~ (counter, label: ~action~) — action
  failures, labeled by action description.
- ~proc_siding_pressure_sample~ (gauge) — most recent sampled pressure value
  from the detector.

Implementation steps:

1. Add ~prometheus = "0.13"~ to workspace dependencies.
2. Add a ~metrics.rs~ in ~crates/lib/src/~ defining a ~Metrics~ struct holding
   the registry and counters/gauges, with a constructor that registers them.
3. Thread ~Metrics~ into ~Monitor~ and bump counters on transitions and action
   errors.
4. Add a ~metrics_server.rs~ in ~crates/lib/src/~ that binds a TCP listener and
   serves ~/metrics~ using ~TextEncoder~.  Keep it simple — raw
   ~std::net::TcpListener~ in a spawned thread is fine; no need for axum.
5. Add ~metrics_listen: Option<String>~ to ~AppConfig~.  When set, spawn the
   metrics server thread in ~main.rs~.
6. Add integration tests: start the metrics server, trigger transitions via a
   mock monitor, scrape ~/metrics~, and assert counter values.

* TODO Make the library resource-agnostic by extracting GPU detectors

~crates/lib~ currently embeds three GPU-specific detector implementations and a
~DetectorConfig~ enum that names GPU vendors (~Amd~, ~Nvidia~, ~Metal~).  This
couples the library to a specific resource type.  The ~PressureDetector~ trait
itself is already resource-agnostic — only the config and implementations are
not.

** Goal

The library (~crates/lib~) should have no knowledge of GPUs.  GPU detection
becomes one possible set of detectors provided alongside the library, not baked
into it.

** Approach: new ~crates/gpu-detectors~ crate

Create a new workspace member ~crates/gpu-detectors~ that holds the three
existing GPU detector structs (~AmdGpuDetector~, ~NvidiaGpuDetector~,
~MetalGpuDetector~), the Linux helpers (~pid_has_device_fd~,
~all_device_users~, ~filter_contributors~), the ~metalps~ serde types, and all
GPU-specific ~DetectorError~ variants.

This crate depends on ~proc-siding-lib~ for the ~PressureDetector~ trait and
~Pid~ type.  The CLI crate depends on both ~proc-siding-lib~ and
~proc-siding-gpu-detectors~.

*** ~crates/lib~ changes

- ~detector.rs~ retains only:
  - ~Pid~ type alias.
  - ~PressureDetector~ trait.
  - A generic ~DetectorError~ enum with resource-agnostic variants (e.g.
    ~SubprocessFailed~, ~OutputParseFailed~, ~SysfsRead~).
  - The ~filter_contributors~ tests (they test a general pattern; the function
    itself moves but the tests can exercise it through the gpu-detectors crate,
    or the function can stay in lib as a public utility since baseline/excluded
    filtering is resource-agnostic).
- ~config.rs~: remove ~DetectorConfig~ entirely.  Detection config moves to the
  CLI layer since it is the wiring layer.
- ~lib.rs~: re-export trait and types only.

*** ~crates/gpu-detectors~ changes

- New crate: ~proc-siding-gpu-detectors~.
- Contains ~amd.rs~, ~nvidia.rs~, ~metal.rs~ with the current detector
  implementations (moved from ~detector.rs~).
- Contains ~helpers.rs~ with ~pid_has_device_fd~, ~all_device_users~,
  ~filter_contributors~ (Linux helpers).
- Owns a ~GpuDetectorConfig~ enum:

** Intel Arc

The Intel Arc discrete GPU uses the ~i915~ (or ~xe~) kernel driver on Linux.
Utilization can be read from:

- ~intel_gpu_top -J~ (from ~intel-gpu-tools~) — JSON output with per-engine
  utilization.
- ~/sys/class/drm/card*/gt/gt*/rc6_residency~ — inverse of busy (residency in
  idle state); would need the complement.

We have no Intel Arc hardware to test on.  Add a note in the GPU detectors
documentation indicating that an Intel Arc detector is welcome but untested.
The ~Exec~ detector (once implemented) would let Intel Arc users bridge the gap
immediately by wrapping ~intel_gpu_top~.
