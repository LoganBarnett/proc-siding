* IN-PROGRESS Replace built-in detectors with exec-based polling

proc-siding is not a detection mechanism — it is a trigger that acts when
pressure is detected.  Detection is an external concern.  The library currently
embeds three GPU-specific detector implementations, a ~PressureDetector~ trait,
a ~ProcessDiscovery~ trait, and config enums that name GPU vendors and discovery
strategies.  All of this should be replaced with a single, simple model:
proc-siding runs a user-configured command at a regular interval and parses
its output.

** Model

The configured command is a *resource poller*.  It runs, inspects whatever
resource it cares about, and writes rows to stdout.  Each row is a
tab-separated pair:

#+begin_example
<value>\t<entity>
#+end_example

- ~value~: a numeric pressure reading (e.g. GPU utilization percentage).
- ~entity~: a human-readable label for the source of the pressure (e.g. a
  process name).  Used only for logging — proc-siding does not interpret it.

proc-siding:

1. Runs the command every ~poll_interval_ms~ (already in ~PressureConfig~).
2. Parses stdout into ~(f64, String)~ rows.
3. Sums the values to produce the aggregate pressure reading.
4. Collects entity names as the contributor list (for log output).
5. Applies hysteresis against the threshold.
6. Fires actions on state transitions.

The command is responsible for all filtering, exclusion, child-process
handling, and resource-specific logic.  proc-siding does not know or care what
resource is being monitored, what PIDs exist, or how exclusion works.

** What gets removed from the library

- ~detector.rs~: delete ~AmdGpuDetector~, ~NvidiaGpuDetector~,
  ~MetalGpuDetector~, all Linux helpers (~pid_has_device_fd~,
  ~all_device_users~, ~filter_contributors~), ~metalps~ serde types, and the
  ~PressureDetector~ trait.  The trait is no longer needed — there is exactly
  one detection strategy (run a command).
- ~discovery.rs~: delete entirely.  ~SystemdUnitDiscovery~, ~PidDiscovery~,
  ~ProcessNameDiscovery~ all move into the user's poller script if needed.
- ~config.rs~: remove ~DetectorConfig~, ~ProcessDiscoveryConfig~.  Replace
  with a ~detector_cmd~ field (the command string to run).
- ~monitor.rs~: simplify — no trait objects for detector or discovery.  The
  monitor runs the command, parses the output, and does hysteresis + actions.

** What stays

- ~action.rs~, ~PressureAction~ trait, ~ActionConfig~ — unchanged.
- ~monitor.rs~ — simplified but same hysteresis state machine.
- ~config.rs~ — ~PressureConfig~ (threshold, hysteresis, ~poll_interval_ms~),
  ~ActionConfig~, ~AppConfig~.
- ~metrics.rs~, ~metrics_server.rs~ — unchanged.

** Example poller scripts (~examples/detectors/~)

The existing Rust detector implementations encode real operational knowledge —
sysfs paths, ~nvidia-smi~ flags, ~metalps~ JSON structure, baseline PID
filtering, ~/proc/<pid>/fd~ scanning, cgroup PID enumeration, ~pgrep~ child
walks.  This knowledge should be preserved as substantial, well-commented
shell scripts in ~examples/detectors/~.  Each script is directly usable as a
~detector_cmd~ value.

*** ~amd-gpu.sh~

- Reads ~gpu_busy_percent~ from ~/sys/class/drm/card*/device/gpu_busy_percent~.
- Scans ~/proc/<pid>/fd~ symlinks for ~/dev/dri/~ to find processes with DRI
  device files open.
- Captures baseline PIDs on first run (processes that had DRI fds open at
  startup, e.g. compositor) and excludes them on subsequent runs.  Uses a
  state file (e.g. ~/run/proc-siding/baseline-pids~) to persist across
  invocations.
- Accepts ~--exclude-unit <unit>~ to read cgroup PIDs from
  ~/sys/fs/cgroup/system.slice/<unit>/cgroup.procs~ and exclude them.
- Only reports pressure when at least one non-excluded, non-baseline process
  has the device open — suppresses the sysfs utilization figure otherwise.
- Outputs TSV: one row per contributing process.
- Requires root or ~CAP_SYS_PTRACE~ for ~/proc/<pid>/fd~ scanning.

*** ~nvidia-gpu.sh~

- Queries ~nvidia-smi --query-gpu=utilization.gpu --format=csv,noheader,nounits~.
- Scans ~/proc/<pid>/fd~ for ~/dev/nvidia~ to find contributing processes.
- Same baseline PID and ~--exclude-unit~ logic as the AMD script.
- Requires ~nvidia-smi~ on ~PATH~ and root for fd scanning.

*** ~metal-gpu.sh~

- Runs ~metalps --json --interval-ms <N>~ and parses the JSON output with ~jq~.
- Sums ~gpu_percent~ across non-excluded processes (capped at 100).
- Accepts ~--exclude-pattern <pattern>~ for ~pgrep -f~ based exclusion
  (plus recursive child walk via ~pgrep -P~).
- No baseline PID logic needed — ~metalps~ reports per-process utilization
  directly.
- Requires ~metalps~ and ~jq~ on ~PATH~.  Runs at user level.

*** Intel Arc (note only)

We have no Intel Arc hardware to test on.  Include a ~README~ or header

driver, and that a wrapper script following the same TSV contract would work.
Contributions welcome.
